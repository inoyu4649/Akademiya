GMCAuto 2의 대규모 업데이트인 GMCAuto 3을 구현한다.

최우선 원칙

작업을 시작하기 전에 반드시 메모리와 코드베이스 전체를 충분히 탐색하여 현재 구조를 정확히 파악할 것.
구현 전 관련 파일과 의존성을 먼저 조사한 뒤 작업할 것.
기존 동작을 추측하여 구현하지 말고 실제 코드베이스를 기준으로 수정할 것.
동일한 기능이 이미 존재하면 재구현하지 말고 최대한 재사용할 것.
불필요한 리팩토링, 네이밍 변경, 코드 스타일 변경은 하지 말 것.
요구사항에 없는 기능 추가 및 삭제를 하지 말 것.
Akademiya 프로젝트와 GMCAuto 프로젝트의 경계를 명확히 유지할 것.
작업 완료 후 변경사항이 요구사항과 일치하는지 자체 검증할 것.
0. 버전 관리
  0-1. 프로젝트 전체에서 버전 배지와 내부 버전을 모두 v2.9.1 -> v3.0으로 변경할 것. 하드코딩, 상수, package 정보 등 사용자에게 노출되는 모든 버전을 함께 확인할 것.
  0-2. UI 구조가 좌측 메뉴 방식으로 변경되는 만큼, 현재 모바일 UI처럼 좌측 메뉴 최상단에는 GMCAuto 로고와 버전 배지만 표시할 것. 기존의 "GMCAuto" 텍스트는 제거할 것.
1. 기존의 Akademiya와 완전 독립된, 독립 GMCAuto 앱 구조는 유지할 것.
  1-1. GMCAuto의 기존 파란 계열 브랜드 테마 색상은 그대로 유지할 것.
  1-2. 단, 색상을 제외한 최신 Akademiya의 디자인 시스템(UI 컴포넌트, 레이아웃, 인터랙션, 스타일 구조)을 최대한 동일하게 반영할 것.
  1-3. 기존의 메타태그, OG Tag, SEO 관련 설정은 모두 유지할 것.
  1-4. HTML Title만 "GMCAuto 3 — Going HAFS 자동신청"으로 변경할 것.
  1-5. Akademiya 프로젝트의 코드는 절대 수정하지 말 것. 필요한 경우 GMCAuto에서만 구현하거나 공용 인터페이스만 사용할 것.
2. 인증 시스템 - 2.9에서 구현 완료 상태, 무시할것.
  2-1. 기존 GMCAuto 전용 oauth.ts 구현을 제거하고 Akademiya OpenOAuth 기반으로 전환할 것.
  2-2. 인증 관련 설정은 .env를 사용할 것.
  2-3. 인증 구현 시 기존 OpenOAuth 구조를 그대로 따르며 GMCAuto에서 별도의 인증 로직을 중복 구현하지 말 것.
3. 계정 시스템
  3-1. 기존 Akademiya 계정 / GMCAuto 계정 이원화 시스템을 제거하고 "Akademiya로 로그인" 버튼 하나만 제공할 것. 인증은 Akademiya OpenOAuth만 사용한다. Google 로그인 및 @hafs.hs.kr Workspace 도메인 판별은 Akademiya OpenOAuth가 담당하므로 GMCAuto에서는 절대 구현하지 말 것.
로그인 버튼 아래에는 작은 회색 글씨로
"Akademiya를 경유해 학교 Google 계정으로 로그인합니다."
를 표시할 것.
  3-2. 로그인 화면 UI를 전면 개편할 것.
Akademiya OpenOAuth의 "Google로 로그인" 버튼 디자인을 참고하여 "Akademiya로 로그인" 버튼으로 변경할 것.
로그인 블록 최하단에는 연파랑 Box를 추가할 것.

이 문구는 3.1 버전에서 삭제 예정이므로 i18n을 사용하지 말고 하드코딩할 것.
  3-3. Akademiya OpenOAuth 구현 시
"C:\Users\이민기\Desktop\개발\Akademiya\guides\akademiya-openoauth-guide.md"
문서를 반드시 참고하여 구현할 것.
문서와 현재 코드가 다를 경우 현재 코드베이스를 우선 기준으로 삼을 것.
4. DB 시스템
  4-1. 기존 GMCAuto perm 시스템은 GMCAuto 계정 기준으로 구현되어 Akademiya 계정에서는 정상 동작하지 않는 문제가 있었다.
GMCAuto 3부터는 Akademiya 계정만 사용하므로 기존 학번 기반 권한 관리를 제거하고 이메일을 Primary Identifier로 사용하여 권한을 관리할 것.
단, 사용자에게 표시되는 학번 정보는 유지한다.
  4-2. GMCAuto 3은 2학기 전환과 함께 DB를 새로 사용하는 구조이므로 기존 DB와의 호환성은 고려하지 않아도 된다.
대신 최초 1회 실행 가능한 Migration Script를 제작할 것.
대상:
- 기존 GMCAuto 2 DB
조건:
- 2026년 6월 17일 신청 결과가 "성공"인 사용자
이식 대상:
- credentials
- 2026년 6월 17일 신청 내역
대상 DB:
- gmcauto3
Migration Script는 재실행 시 중복 이식되지 않도록 안전하게 작성할 것.
  4-3. 관리자 대시보드 통계에서는 기존과 동일하게 학번을 표시할 것.
  4-4. 기존 자정 복사 방식의 일정 관리 로직은 제거하고 DB 기반 일정 관리 시스템으로 변경할 것.
일정 조회 및 신청 가능 여부는 항상 DB 기준으로 판단한다.
이를 통해 중단 기간 변경 시 기존처럼 하루가 추가로 열리는 문제를 해결할 것.
5. UI/UX
  5-1. 예약 현황 표시 방식을 전면 개선할 것.
기존처럼 예약자별 빨간 블록을 나열하지 말고,
30분 단위를 한 줄로 구성하여 시간축 기반으로 표시할 것.
색상 규칙은 다음과 같다.
- 예약된 구간: 빨간색
- 자신의 예약: 파란색
- 빈 구간: 회색
모바일 환경을 고려하여 각 분 단위의 폭은 매우 좁게 구성하되,
경계선은 명확하게 표시할 것.
Hover 시 정확한 시:분 정보를 Tooltip으로 표시할 것.
  5-2. 메인 페이지 상단의 "2학기 계정 전환" 경고는 삭제할 것.
  5-3. 기존 기능이 있다면 동작은 그대로 유지하되 UI만 새로운 디자인에 맞게 변경할 것.
  5-4. Akademiya와 동일한 좌측 메뉴 구조를 적용할 것.
좌측 메뉴 최상단에는 Akademiya 로고 대신 GMCAuto 로고를 사용한다.
언어 선택, 라이트/다크 모드, 알림 영역은 Akademiya 구조를 그대로 사용할 것.
메뉴 그룹 구성은 Akademiya의 "개발자 도구" 방식처럼 구성한다.
현재 그룹은 다음과 같다.
- GMC PASS
- 관리자 도구
향후 추가 예정인 "정기외출" 그룹은 현재 구현하지 말 것.
  5-5. 전반적인 테마, 디자인은 기존의 Akademiya-blueified 대신 Akademiya와 조금 멀어져서 C:\Users\이민기\Desktop\개발\GMCRedirectServer\index.html 스타일의 현행 GMCAuto 2보다 조금 더 부드럽고 자연스러운 스타일을 추구할 것.
6. API
  6-1. GMCAuto API를 구축할 것.
향후 Akademiya 및 기타 서비스에서 학교 홈페이지 동시접속으로 인한 차단을 방지하기 위한 공용 API로 활용 가능하도록 설계할 것.
최소 제공 정보는 다음과 같다.
- 학번 기준 신청 여부(Boolean)
- 예약 시간
- 신청 내역
API는 향후 확장을 고려하여 버전 관리와 응답 구조를 일관성 있게 설계할 것.

7. 최종 작업 체크리스트

모든 요구사항이 구현되었는지 확인할 것.
기존 기능이 의도치 않게 변경되지 않았는지 확인할 것.
TypeScript 타입 오류가 없는지 확인할 것.
Lint 오류가 없는지 확인할 것.
Build가 정상 완료되는지 확인할 것.
사용하지 않는 코드, import, 변수는 제거할 것.
요구사항에 없는 변경사항은 포함하지 말 것.
작업 완료 후 변경된 파일과 주요 변경사항을 요약하여 보고할 것.
