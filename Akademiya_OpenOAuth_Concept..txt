Akademiya OpenOAuth 기능을 추가해줘.

1. 기존에 존재하는 "GMCAuto 2 OAuth" 기능을 기반으로 해줘 (절대 AkashaAlt 쪽 참조 금지)
  1-1. 기존 기능을 수정하여 동작이 변경되지 않도록 한다.
  1-2. 신규 기능은 기존 시스템과 자연스럽게 통합한다. 모든 UI스타일은 통일해야 한다. 메모리와 코드베이스를 참고하라.
2. 회원정보 수정 메뉴에 '개발자 모드' 체크박스를 추가해주고, 개발자 모드가 활성화되면 좌측바에 '개발자 도구'가 뜨게 해줘.
3. 개발자 도구 안에 여러 메뉴를 추가할 수 있되, 첫 메뉴로 'Akademiya OAuth'를 추가해줘.
4. Akademiya OpenOAuth API 설정은 사용자가 다음과 같이 할 수 있게 해줘.
  4-1. 로그인 허용 범위: 전체, 조직, 반 단위, Google Workspace(Google 전용인 경우에만 선택 가능)
  4-2. 로그인 허용 수단: Akademiya 전용/Google 전용/둘 다 허용 (여기서 Google은 Akademiya에 Google로 로그인하는 기능을 말함, 별도의 Google OAuth를 구현하는 것이 아니다.)
  4-3. 내부 코드명(예: akademiya-oauth-example: 영문 숫자 하이픈만 허용, 생성 후 변경 불가.)과 외부 표출 이름(예: Akademiya OAuth Example, OAuth 승인 화면에 표시되는 이름이다. 수정 가능하다.)을 모두 입력하게.
  4-4. 통계 조회 기능: 자신의 OAuth App에 로그인한 사용자 수, 로그인 요청 건수 (기간별 통계). 기간 선택: 오늘, 최근 7일, 최근 30일, 사용자 지정 기간. 그래프 및 숫자를 함께 제공한다.
  4-5. OAuth Ban 기능: 특정 사용자를 BAN할 수 있게. 사용자 검색, BAN, BAN 해제, BAN 사유, BAN 일시. BAN된 사용자가 OAuth 로그인을 시도하면 적절한 오류 메시지를 표시한다.
5. API키는 첫 발급 시에만 확인 가능하고, 확인 클릭 이후에는 재발급해야만 하게 해줘.
  5-1. OAuth App 생성 시 Client ID, Client Secret을 발급한다.
  5-2. Google Cloud Console과 동일한 UX를 따른다.
6. 구글처럼 OAuth App URL 허용(신뢰 JavaScript 출처) 목록을 추가해야 사용할 수 있게 해줘 (화이트리스트 제도) 등록되지 않은 Origin에서는 OAuth 요청을 거부한다.
  6-1. 이 외에도 OAuth App의 메인 사이트(예: example.akademiya.kr) URL도 딱 하나 입력하게. (1개만 허용)
7. 사용자의 OAuth 로그인 화면은 다음과 같게.
  7-1. 기본적으로 Akademiya 로그인 화면의 UI를 기반으로 한다. 디자인은 최대한 동일하게 유지한다.
  7-2. 로그인 허용 수단에 따라 화면을 구성해줄 것: Akademiya 전용일 경우 Google로 로그인 버튼 안 보이게, Google 전용일 경우 Akademiya 이메일 비번 로그인 버튼과 회원가입 버튼 모두 안 보이게, 둘 다 허용인 경우 Akademiya 로그인 화면과 같게
  7-3. 로고 밑 '로그인' 텍스트 바로 하단에 강조색 중 박스에 사용할 색으로(css 참고.) 박스를 통해 현재 로그인하고 있는 대상 OAuth App을 표시해줘. (예: Akademiya OAuth Example에 로그인), 여기서 저 OAuth App 이름은 메인 사이트 링크(URL)를 걸어줘.
8. 보안 사항 (매우 중요)
  8-1. 다음 사항을 반드시 구현한다.
OAuth Origin Whitelist 검증
Redirect URI 검증
OAuth State 검증
CSRF 방어
PKCE 지원
Client Secret 안전한 저장(평문 저장 금지)
Access Token/Refresh Token 발급
권한 범위(Scope) 검증
승인되지 않은 Origin 차단
BAN 사용자 로그인 차단
9. UI/UX
  9-1. 기존 Akademiya 디자인 시스템을 따른다.
  9-2. 기존 CSS 컴포넌트를 최대한 재사용한다.
  9-3. 새로운 디자인을 만들기보다 기존 스타일을 활용한다.
  9-4. 기존 Akademiya 방식대로 i18n과 다크 모드를 지원한다.
  9-5. 모바일에서도 정상 동작한다.
10. 코드 품질
  10-1. 기존 프로젝트 구조를 유지한다.
  10-2. 중복 코드를 만들지 않는다.
  10-3. 공통 컴포넌트를 적극 활용한다.
  10-4. TypeScript 타입 안정성을 유지한다.
  10-5. 불필요한 리팩터링은 하지 않는다.
  10-6. 기존 기능에 영향을 주지 않는다.
